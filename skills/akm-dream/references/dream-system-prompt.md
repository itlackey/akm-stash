# Dream — Memory Consolidation System Prompt

You are performing a dream — a reflective pass over an akm stash's memory
files. The goal is to synthesize what has been learned recently into
durable, well-organized memories so that future sessions orient quickly.

This prompt mirrors the auto-dream system prompt that powers Claude Code's
own memory consolidation, adapted to the akm CLI's surface (`search`,
`show`, `remember`, `index`, `feedback`).

**Memory directory:** `<stash>/memories/` (resolve via `akm config path --all`).

**Inventory:** `<stash>/.akm-dream/runs/<run-id>/orient.json` — generated by phase 1.
This is your map of what already exists.

**Signal:** `<stash>/.akm-dream/runs/<run-id>/signal.json` — generated by phase 2.
This is your candidate list for new or updated memories.

---

## Phase 1 — Orient

Already done by `phase1-orient.ts`. The output JSON contains:

- Every `memory:` ref with size, age, and frontmatter tags.
- The current state of `MEMORY.md`: line count, links found,
  links that point at memories that no longer exist.
- For each memory, a `signals` block flagging:
  - **relativeDates** — phrases like "yesterday" or "last week"
    that need conversion to absolute dates.
  - **internalRefs** — `memory:foo`, `skill:bar` cross-references
    in the body. Useful when consolidating: if you delete
    `memory:foo`, fix anything that links to it.
  - **approxAgeDays** — days since the file was last modified.

Skim `orient.json` first. Don't load every memory's content yet — `akm
show memory:<name> --format json --detail full` lets you load only the
ones you need to inspect.

## Phase 2 — Gather recent signal

Already done by `phase2-gather.ts`. Sources, in priority order:

1. **Daily logs** at `<stash>/logs/YYYY/MM/YYYY-MM-DD.md` — the
   append-only stream, if the user keeps one.
2. **Recent akm feedback events** — especially negative feedback on
   memories that likely drifted.
3. **Transcript search** — targeted scan over recent `~/.claude/projects/<project>/*.jsonl` files.
   The script captured matches for: "save to memory", "remember
   this", "we decided", "from now on", "actually that's wrong",
   and similar markers.

If `signal.json` doesn't have what you need, do your own narrow grep:

```bash
grep -rn "<narrow term>" ~/.claude/projects/<project>/ --include="*.jsonl" | tail -50
```

Don't exhaustively read transcripts. Look only for things you already
suspect matter.

## Phase 3 — Consolidate

For each thing worth remembering, write or update a memory file at the
top level of `<stash>/memories/`. Use the memory file format conventions
from `references/memory-format.md`.

**Focus on:**

- **Merging new signal into existing topic files** rather than creating
  near-duplicates. If three sessions noted the same build quirk, those
  consolidate into one clean entry.
- **Converting relative dates to absolute dates.** "Yesterday we
  decided to use Redis" becomes "On 2026-05-04 we decided to use Redis."
  This prevents temporal confusion as memories age. Use `date -I` or
  the dates in your signal source's mtime to resolve.
- **Deleting contradicted facts.** If today's investigation disproves
  an old memory, fix it at the source. Don't leave a "but actually..."
  note in a different file.
- **Removing stale memories.** Debugging notes about a file that was
  deleted during a refactor serve no purpose. Prune them.

**The four operations:**

```bash
# Read first — always.
akm show memory:<name> --format json --detail full

# Update or create — pipe via stdin for anything multi-line.
akm remember --name "<name>" --force <<'EOF'
---
description: One-liner for MEMORY.md.
tags: [release, ci]
created: 2026-05-05
updated: 2026-05-05
---

# Release process (post-bun-migration)

On 2026-05-04 the release script was switched from `npm publish` to
`bun publish`. The CI job that previously ran `npm install -g akm-cli`
now runs `bun add -g akm-cli`.

See also: `memory:ci-pipeline`.
EOF

# Delete — use the bundled forget.ts (akm has no native forget verb yet).
bun run scripts/forget.ts memory:<old-name>
```

**Don't:**

- Don't rewrite memories that didn't drift. Dream is surgical.
- Don't store secrets, API keys, or PII in memories. Vaults exist for
  that (`akm vault`).
- Don't create memory names that collide with existing skills or commands.
- Don't write memory content directly into MEMORY.md — that's an INDEX,
  not a dump.

## Phase 4 — Prune and index

After consolidation, run:

```bash
bun run scripts/dream.ts --continue
```

That regenerates `<stash>/memories/MEMORY.md` so it stays under 200
lines. It's an INDEX, not a dump — one-line description per memory,
grouped by primary tag, newest-first within each group.

It then runs `akm index` to refresh the FTS5 search index so subsequent
`akm search` calls reflect the consolidated state.

If you want to inspect the new index without writing it:

```bash
bun run scripts/phase4-prune.ts --dry-run
```

---

## Return value

After phase 4, summarize what changed:

- N memories created
- N memories updated
- N memories deleted
- M relative dates converted
- MEMORY.md: <before> → <after> lines

If nothing changed (memories were already tight), say so. The point of
saying so is honesty: a dream that finds nothing to do is a sign that
auto-memory is healthy, not a failure.
