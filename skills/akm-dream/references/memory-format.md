# akm Memory File Format

akm stores memories as plain markdown files under `<stash>/memories/`,
one file per memory. The dream pipeline reads, writes, and deletes
files following the conventions below.

## File layout

```
<stash>/
└── memories/
    ├── MEMORY.md              # the index (regenerated by phase 4)
    ├── release-process.md
    ├── ci-pipeline.md
    └── ops/
        └── prod-secrets.md   # nested names are allowed
```

The filename (minus `.md`) is the memory's `name`. With `akm remember
--name release-process`, the file is `memories/release-process.md` and
the ref is `memory:release-process`. Names containing `/` create
subdirectories.

## Frontmatter

Frontmatter is YAML between two `---` lines. All fields are optional
but the dream pipeline uses them when present:

```markdown
---
description: One-liner shown in MEMORY.md.
tags: [release, ci]
created: 2026-04-12
updated: 2026-05-05
---
```

| Field         | Purpose                                                    |
| ------------- | ---------------------------------------------------------- |
| `description` | Used by `phase4-prune` to populate the MEMORY.md index. If absent, the first sentence of the body is used. |
| `tags`        | Primary tag (first one) becomes the MEMORY.md section. Untagged memories land under "uncategorized". |
| `created`     | ISO date of original creation. Useful when the file's mtime has been touched by other tools. |
| `updated`     | ISO date of last meaningful update. Lets dream tell stale memories from fresh ones across mtime resets. |

The dream pipeline tolerates frontmatter-less memories — it falls back
to file mtime for age and the first sentence for description. New
memories created during phase 3 should include frontmatter so future
dream cycles can do better work.

## Body conventions

- **First heading**: a single `#` H1 with the memory's title. The H1
  text is human-readable and may differ from the slug-style filename.
- **Use absolute dates.** "On 2026-05-04 we decided X." Not "yesterday".
- **Cross-reference other assets** with akm refs:
  ```
  See also: `memory:ci-pipeline`, `skill:deploy`.
  ```
  These are picked up by `scanMemorySignals` and used to keep the
  dependency graph consistent during consolidation.
- **One topic per file.** If a memory drifts into covering two topics,
  split it during the next dream cycle.
- **No secrets.** Use vaults (`akm vault`) for anything sensitive.

## Example memory

```markdown
---
description: Release scripts now run via bun publish, not npm publish.
tags: [release, ci]
created: 2026-04-12
updated: 2026-05-04
---

# Release process

On 2026-05-04 the release script was switched from `npm publish` to
`bun publish`. The CI job that previously ran `npm install -g akm-cli`
now runs `bun add -g akm-cli`.

The change was driven by:

- 30% faster install on CI runners
- npm's lockfile drifting against bun.lock
- bun's built-in workspaces resolution matching our monorepo layout

If a release fails with `EACCES: permission denied`, the CI runner
likely doesn't have a global bun install yet. Fix with:

    curl -fsSL https://bun.sh/install | bash

See also: `memory:ci-pipeline`, `skill:release`.
```

## What dream looks for

Phase 1 (`scanMemorySignals`) flags:

- **Relative-date phrases** — "yesterday", "last week", "X days ago",
  "recently", "the other day". These need rewriting to absolute dates
  during phase 3.
- **Internal refs** — every `<type>:<name>` mention is recorded so
  phase 3 can keep the cross-references consistent when memories
  are merged or deleted.
- **External URLs** — bare `https?://...` links. Surfaced for the
  agent so it can decide whether they should be archived as
  knowledge assets (`akm import`) instead.

Phase 4 uses `description`, `tags`, and mtime to render `MEMORY.md`.
A memory without frontmatter still appears in the index — it just gets
a less informative entry.
